# MATH.md — Математика внутри `recognition.py`, подробно и со ссылками на код

## TL;DR

Вся логика файла сводится к такой цепочке:

```text
изображение → лицо → фильтрация качества → embedding ∈ ℝ^128 → попарные расстояния → nearest neighbor → threshold
```

То есть здесь нет «магии распознавания лиц» на уровне основного пайплайна.  
После получения face embedding всё становится обычной геометрией в многомерном пространстве:

- лицо = точка;
- набор селфи = облако точек;
- совпадение = ближайшая точка находится достаточно близко.

# 3. Ресайз: линейное масштабирование координат

Код:  
<https://github.com/pavelveter/recognition/blob/main/recognition.py#L140-L148>

```python
h, w = image.shape[:2]
mh, mw = max_hw
scale = min(mh / h, mw / w)
if scale < 1.0:
    new_size = (int(w * scale), int(h * scale))
    return cv2.resize(image, new_size, interpolation=cv2.INTER_LINEAR)
```

## Что происходит математически

Используется коэффициент:

```text
scale = min(mh / h, mw / w)
```

После этого каждая координата масштабируется одинаково:

```text
(x, y) → (scale · x, scale · y)
```

Это обычное линейное преобразование в 2D.

## Почему это важно

Если масштабировать по разным коэффициентам по X и Y, лицо деформируется.  
Здесь коэффициент один и тот же, поэтому пропорции сохраняются.

## Пример

Пусть есть фото:

```text
4000 × 3000
```

Нужно вписать в:

```text
1600 × 1600
```

Тогда:

```text
1600 / 4000 = 0.4
1600 / 3000 ≈ 0.5333
scale = 0.4
```

Новый размер:

```text
(3000 · 0.4, 4000 · 0.4) = (1200, 1600)
```

## Что значит bilinear interpolation

`cv2.INTER_LINEAR` — это билинейная интерполяция.

Интуитивно:
- новый пиксель получается как взвешенная комбинация соседних пикселей;
- это сглаживает пересчёт и уменьшает артефакты.

То есть даже на этапе ресайза уже есть локальная линейная модель.

# 4. Отсечение слишком маленьких лиц

Код:
- вычисление `min_face_size`:  
  <https://github.com/pavelveter/recognition/blob/main/recognition.py#L68-L72>
- проверка в `is_face_clear`:  
  <https://github.com/pavelveter/recognition/blob/main/recognition.py#L154-L157>

```python
min_face_size: int = int(max(max_size_tuple) / 33)
...
face = image_bgr[top:bottom, left:right]
h, w = face.shape[:2]
if h < min_face_size or w < min_face_size:
    return False
```

## Что это означает

Здесь используется простая геометрическая отсечка:

- если прямоугольник лица слишком мал,
- лицо сразу считается непригодным.

## Математический смысл

У маленького лица:
- слишком мало пикселей,
- производные становятся шумными,
- частотная картина искажается,
- embedding становится менее устойчивым.

То есть проверка не про красоту кода, а про область применимости последующей математики.

# 5. Преобразование в grayscale: линейная комбинация каналов

Код:  
<https://github.com/pavelveter/recognition/blob/main/recognition.py#L159-L159>

```python
gray = cv2.cvtColor(face, cv2.COLOR_BGR2GRAY)
```

## Что происходит математически

Цветное изображение состоит из трёх каналов: `B`, `G`, `R`.

Переход в серый — это взвешенная линейная комбинация:

```text
Gray = αB + βG + γR
```

Точные коэффициенты определяются библиотекой, но смысл именно такой:
из трёх чисел на пиксель получается одно число яркости.

## Зачем это нужно

Для:
- градиентов,
- Laplacian,
- FFT,
- LBP

цвет не так важен, как структура яркости.

То есть дальше код работает не с цветом, а с сигналом яркости.

# 6. Laplacian: вторая производная изображения

Код:  
<https://github.com/pavelveter/recognition/blob/main/recognition.py#L161-L163>

```python
lap_var = float(np.var(cv2.Laplacian(gray, cv2.CV_64F)))
if lap_var >= laplacian_threshold:
    return True
```

## Формула

Для изображения `I(x, y)` Лапласиан:

```text
∇²I = ∂²I/∂x² + ∂²I/∂y²
```

В коде используется дискретная версия этого оператора.

## Интуиция

Первая производная показывает скорость изменения.  
Вторая — насколько резко меняется сама скорость изменения.

На резких границах лица:
- контуры глаз,
- ресницы,
- граница губ,
- линия носа,

значения второй производной обычно заметнее.

На мыльном изображении переходы размазаны, поэтому значения более сглажены.

## Почему берётся дисперсия

Используется:

```text
var(Laplacian(gray))
```

Дисперсия показывает разброс значений.

- большая дисперсия → в изображении много резких локальных изменений;
- маленькая дисперсия → лицо гладкое, вероятно, размытое.

## Пример

Если Лапласиан по пикселям даёт примерно:

```text
[-18, 24, -12, 30, -21]
```

разброс большой → дисперсия большая.

Если:

```text
[-1, 0, 1, 0, -1]
```

разброс маленький → дисперсия маленькая.

## Особенность логики

Здесь используется ранний выход:

```python
if lap_var >= laplacian_threshold:
    return True
```

То есть если дешёвая метрика уже говорит «лицо достаточно чёткое», дальнейшие, более дорогие проверки можно не выполнять.

# 7. Sobel и градиент: первая производная изображения

Код:  
<https://github.com/pavelveter/recognition/blob/main/recognition.py#L165-L169>

```python
sobel_x = cv2.Sobel(gray, cv2.CV_64F, 1, 0, ksize=3)
sobel_y = cv2.Sobel(gray, cv2.CV_64F, 0, 1, ksize=3)
grad_mean = float(np.mean(np.sqrt(sobel_x**2 + sobel_y**2)))
if grad_mean >= gradient_threshold:
    return True
```

## Формулы

Смысл Sobel:

```text
I_x ≈ ∂I/∂x
I_y ≈ ∂I/∂y
```

Дальше вычисляется модуль градиента:

```text
|∇I| = sqrt(I_x² + I_y²)
```

Это обычная евклидова длина двумерного вектора `(I_x, I_y)`.

## Интуиция

Градиент говорит, насколько быстро меняется яркость.

Если в лице много деталей и контраста:
- градиент большой.

Если изображение замылено:
- яркость меняется медленно;
- градиент в среднем меньше.

## Почему берётся среднее

Код использует:

```text
mean(|∇I|)
```

Это усреднённая «сила границ» по всей области лица.

## Пример

Если локальные значения `|∇I|` такие:

```text
[2, 5, 8, 6, 4]
```

то среднее:

```text
5.0
```

Если порог 4 — лицо проходит.

Если значения:

```text
[0.3, 0.6, 0.8, 0.2, 0.5]
```

то среднее маленькое — лицо, вероятно, некачественное.

## Почему снова ранний выход

Если по градиенту уже видно, что лицо чёткое, нет смысла считать FFT и LBP.

# 8. High-pass: выделение деталей через вычитание размытой версии

Код:  
<https://github.com/pavelveter/recognition/blob/main/recognition.py#L171-L172>

```python
high_pass = gray - cv2.GaussianBlur(gray, (5, 5), 10)
high_pass_var = float(np.var(high_pass))
```

## Формула

Если обозначить изображение как `I`, а его размытую версию как `G(I)`, то:

```text
I_high = I - G(I)
```

## Интуиция

Размытие оставляет:
- крупные, плавные структуры,
- общий тон,
- медленные изменения.

Вычитание размытой версии оставляет:
- мелкие детали,
- контуры,
- текстуру,
- всё высокочастотное.

## Почему это high-pass

В терминах обработки сигналов:
- blur ведёт себя как low-pass,
- `I - blur(I)` ведёт себя как high-pass.

## Почему берётся дисперсия

Если в high-pass сигнале есть заметная энергия, значит:
- лицо содержит детали.

Если после вычитания почти всё близко к нулю:
- изображение слишком гладкое,
- детализация потеряна.

## Пример

После `I - G(I)` получились значения:

```text
[-12, 9, -8, 11, 6]
```

Разброс большой → деталей много.

Если:

```text
[-1, 0, 1, 0, -1]
```

Разброс маленький → картинка слишком сглажена.

# 9. FFT: переход в частотную область

Код:  
<https://github.com/pavelveter/recognition/blob/main/recognition.py#L174-L177>

```python
fft = np.fft.fft2(gray)
fft_mag = np.abs(np.fft.fftshift(fft))
mask95 = fft_mag > np.percentile(fft_mag, 95)
high_freq_mean = float(np.mean(fft_mag[mask95]))
```

## Что такое FFT здесь

2D FFT переводит изображение из пространственной области в частотную:

```text
I(x, y) → F(u, v)
```

Где:
- `(x, y)` — координаты пикселей;
- `(u, v)` — частотные координаты.

## Интуиция

В частотной области:
- низкие частоты = крупные и плавные изменения;
- высокие частоты = резкие детали, текстуры, границы.

Размытие обычно подавляет именно высокие частоты.

## Что делает код по шагам

1. Считает спектр:
   ```python
   fft = np.fft.fft2(gray)
   ```

2. Сдвигает нулевую частоту в центр:
   ```python
   fftshift
   ```

3. Берёт амплитуду:
   ```python
   abs(...)
   ```

4. Отрезает верхние 5% значений:
   ```python
   fft_mag > percentile(..., 95)
   ```

5. Усредняет эти большие значения.

## Что это даёт

Это не академически строгий тест «именно высоких частот по радиальной маске», а практическая эвристика:

- если в спектре есть заметные крупные компоненты среди верхнего хвоста,
- лицо, вероятно, не совсем замылено.

## Важная оговорка

*NOT SURE*: строго говоря, этот кусок кода измеряет не «чистую энергию высокочастотной области» в строгом спектральном смысле, а усредняет верхний амплитудный хвост после `fftshift`. Это полезная инженерная эвристика, но не каноническая метрика качества изображения.

# 10. LBP: локальная текстура

Код:  
<https://github.com/pavelveter/recognition/blob/main/recognition.py#L179-L182>

```python
radius = 1
n_points = 8 * radius
lbp = local_binary_pattern(gray.astype(np.uint8), n_points, radius, method="uniform")
lbp_mean = float(np.mean(lbp))
```

## Что такое LBP

Для каждого пикселя сравниваются соседи с центральным пикселем:

- сосед ≥ центр → 1
- сосед < центр → 0

Биты собираются в локальный код.

## Интуиция

LBP — это компактный способ описывать локальную текстуру:

- гладкая поверхность,
- границы,
- мелкие повторяющиеся структуры,
- шум.

## Почему это полезно для качества лица

Если лицо:
- слишком размыто,
- слишком шумно,
- потеряло микро-структуру,

распределение LBP-кодов меняется.

Здесь используется не вся гистограмма LBP, а только среднее значение, что грубее, но дешевле.

## Оговорка

*NOT SURE*: само по себе среднее значение LBP не является универсальной и теоретически «чистой» метрикой качества. Здесь это скорее эмпирический фильтр, который имеет смысл только вместе с остальными метриками и конкретными порогами.

# 11. Совместное решение о качестве лица

Код:  
<https://github.com/pavelveter/recognition/blob/main/recognition.py#L184-L193>

```python
if (grad_mean < gradient_threshold or
    high_pass_var < high_pass_threshold or
    high_freq_mean < high_freq_threshold or
    lbp_mean > lbp_threshold):
    ...
    return False
return True
```

## Что это означает

Фактически для каждого лица строится вектор метрик качества:

```text
q = [lap_var, grad_mean, high_pass_var, high_freq_mean, lbp_mean]
```

Дальше вместо обученного классификатора используется rule-based область допустимости.

То есть решение выглядит примерно так:

```text
лицо принимается,
если оно попадает в заданную область допустимых значений
```

## Геометрическая интерпретация

Это можно представить как пространство признаков качества.

Каждое лицо — точка в пространстве:

```text
(lap_var, grad_mean, high_pass_var, high_freq_mean, lbp_mean)
```

Дальше точка либо проходит пороги, либо нет.

# 12. Извлечение embedding: переход от изображения к точке в ℝ^128

Код:
- общий блок:  
  <https://github.com/pavelveter/recognition/blob/main/recognition.py#L197-L239>
- детектор лиц:  
  <https://github.com/pavelveter/recognition/blob/main/recognition.py#L224-L225>
- энкодинг лица:  
  <https://github.com/pavelveter/recognition/blob/main/recognition.py#L229-L235>

Ключевой фрагмент:

```python
face_locs = face_recognition.face_locations(image_rgb, model="hog")
...
face_enc = face_recognition.face_encodings(image_rgb, [(top, right, bottom, left)])
if face_enc:
    encs.append(face_enc[0].astype(np.float32, copy=False))
```

## Что происходит

Сначала находятся координаты лиц.  
Потом для каждого лица извлекается embedding:

```text
x ∈ ℝ^128
```

То есть лицо больше не рассматривается как матрица пикселей.  
Оно становится вектором из 128 чисел.

## Главная идея

Модель `face_recognition` уже переводит лицо в пространство признаков, где:

- лица одного человека должны лежать ближе;
- лица разных людей — дальше.

Текущий файл не обучает это пространство, а только использует его геометрию.

# 13. Почему здесь `float32`

Код:
- преобразование embedding к `float32`:  
  <https://github.com/pavelveter/recognition/blob/main/recognition.py#L235-L238>
- матричные расстояния тоже принудительно переводятся к `float32`:  
  <https://github.com/pavelveter/recognition/blob/main/recognition.py#L542-L543>

## Что это даёт

`float32`:
- дешевле по памяти, чем `float64`;
- обычно достаточно точен для embedding;
- удобен для векторных библиотек и BLAS-подобных операций.

## Но есть цена

`float32` — это приближённая арифметика.  
Из-за этого дальше появляется защита:

```python
np.maximum(d2, 0.0, out=d2)
```

чтобы маленькие отрицательные хвосты после округления не ломали `sqrt`.

# 14. Матрица embeddings: уже не один вектор, а набор

Код:  
<https://github.com/pavelveter/recognition/blob/main/recognition.py#L237-L239>

```python
enc_mat = np.asarray(encs, dtype=np.float32) if encs else np.empty((0, 128), dtype=np.float32)
```

## Что это означает

Если на фото найдено `M` лиц, получается матрица:

```text
A ∈ ℝ^(M×128)
```

где каждая строка — embedding одного лица.

Для набора селфи позже строится похожая матрица:

```text
B ∈ ℝ^(N×128)
```

Это уже чистая линейная алгебра:

- строки = объекты,
- столбцы = признаки.

# 15. Сбор embeddings селфи

Код:  
<https://github.com/pavelveter/recognition/blob/main/recognition.py#L595-L611>

```python
all_selfie_encs: list[np.ndarray] = []
for sf in selfies:
    sp = os.path.join(folder_path, sf)
    encs = load_encodings_for_image(sp, check_quality=False, do_resize=resize_flag)
    all_selfie_encs.extend(encs)

selfie_mat = np.asarray(all_selfie_encs, dtype=np.float32)  # (S,128)
```

## Что это значит

Все embeddings селфи из одной папки объединяются в одну матрицу:

```text
B ∈ ℝ^(S×128)
```

Геометрически это облако точек — эталонное множество лица человека.

Если у одного человека несколько селфи:
- это не один эталон, а набор эталонов,
- что лучше ловит вариации по ракурсу, свету и мимике.

# 16. Сердце всего файла: быстрый расчёт расстояний

Код:  
<https://github.com/pavelveter/recognition/blob/main/recognition.py#L533-L549>

```python
def min_l2_dist_mat(A: np.ndarray, B: np.ndarray) -> np.ndarray:
    if A.size == 0:
        return np.full((0,), np.inf, dtype=np.float32)
    if B.size == 0:
        return np.full((A.shape[0],), np.inf, dtype=np.float32)
    A = A.astype(np.float32, copy=False)
    B = B.astype(np.float32, copy=False)
    aa = np.sum(A*A, axis=1, keepdims=True)    # (M,1)
    bb = np.sum(B*B, axis=1, keepdims=True).T  # (1,N)
    prod = A @ B.T                             # (M,N)
    d2 = aa + bb - 2.0 * prod
    np.maximum(d2, 0.0, out=d2)
    return np.sqrt(np.min(d2, axis=1))
```

Это и есть главный математический блок.

# 17. Квадрат нормы: `aa` и `bb`

Код:
- `aa`:  
  <https://github.com/pavelveter/recognition/blob/main/recognition.py#L544-L544>
- `bb`:  
  <https://github.com/pavelveter/recognition/blob/main/recognition.py#L545-L545>

```python
aa = np.sum(A*A, axis=1, keepdims=True)
bb = np.sum(B*B, axis=1, keepdims=True).T
```

## Формула

Для строки `a_i`:

```text
||a_i||² = Σ_k a_{ik}²
```

Для строки `b_j`:

```text
||b_j||² = Σ_k b_{jk}²
```

## Почему именно квадрат нормы

Потому что квадрат евклидова расстояния удобно раскрывается через:

```text
||a - b||² = ||a||² + ||b||² - 2(a · b)
```

## Размерности

Если:
- `A` имеет форму `(M, 128)`,
- `B` имеет форму `(N, 128)`,

то:
- `aa` имеет форму `(M, 1)`,
- `bb` после транспонирования имеет форму `(1, N)`.

Так они потом красиво броадкастятся в матрицу `(M, N)`.

# 18. Скалярное произведение через матричное умножение

Код:  
<https://github.com/pavelveter/recognition/blob/main/recognition.py#L546-L546>

```python
prod = A @ B.T
```

## Формула

```text
prod[i, j] = a_i · b_j = Σ_k a_{ik} b_{jk}
```

## Что здесь важно

Это не «просто умножение матриц».

Это:
- сразу все попарные скалярные произведения
- между лицами на фото
- и всеми селфи.

То есть вместо двойного цикла:

```python
for a in A:
    for b in B:
        dot(a, b)
```

всё считается одним матричным выражением.

## Пример

Пусть:

```text
A = [[1, 2],
     [3, 4]]

B = [[5, 6],
     [7, 8]]
```

Тогда:

```text
B.T = [[5, 7],
       [6, 8]]
```

и

```text
A @ B.T =
[[1*5 + 2*6, 1*7 + 2*8],
 [3*5 + 4*6, 3*7 + 4*8]]
=
[[17, 23],
 [39, 53]]
```

Каждый элемент — обычный dot product.

# 19. Главная формула: квадрат евклидова расстояния

Код:  
<https://github.com/pavelveter/recognition/blob/main/recognition.py#L547-L547>

```python
d2 = aa + bb - 2.0 * prod
```

## Формула

```text
||a - b||² = ||a||² + ||b||² - 2(a · b)
```

## Вывод

Начинаем с:

```text
||a - b||² = (a - b) · (a - b)
```

Раскрываем:

```text
= a·a - a·b - b·a + b·b
```

Так как `a·b = b·a`, получаем:

```text
= ||a||² + ||b||² - 2(a·b)
```

## Почему это лучше прямого подсчёта

Можно было бы считать так:

```text
Σ_k (a_k - b_k)²
```

для каждой пары `(a, b)` отдельно.

Но форма через:
- нормы
- и матрицу dot product

лучше векторизуется и быстрее работает на NumPy.

## Что получается на выходе

Матрица:

```text
D² ∈ ℝ^(M×N)
```

где

```text
D²[i, j] = ||a_i - b_j||²
```

# 20. Защита от численных ошибок

Код:  
<https://github.com/pavelveter/recognition/blob/main/recognition.py#L548-L548>

```python
np.maximum(d2, 0.0, out=d2)
```

## Зачем это нужно

Теоретически квадрат расстояния не может быть отрицательным.

Но из-за округлений `float32` иногда выходит, например:

```text
-1e-7
```

вместо нуля.

Если сделать:

```python
sqrt(-1e-7)
```

получится `nan`.

Поэтому всё, что меньше нуля из-за численной ошибки, просто обрезается до нуля.

Это очень правильный инженерный ход.

# 21. Минимум по строке = ближайшее селфи

## Что это означает

Это и есть классическая задача nearest neighbor.

```python
dmins = min_l2_dist_mat(E, _WORKER_SELFIE_ENCS)

```

и

```python
dmin = float(np.min(min_l2_dist_mat(E, selfie_mat)))
if dmin <= threshold_v:

```

## Формула

Если для лица на фото `a_i` выполнено:

```text
min_j ||a_i - b_j|| ≤ τ
```

то лицо считается совпавшим.

А для всей фотографии условие становится примерно таким:

```text
если на фото существует хотя бы одно лицо,
у которого ближайшее селфи ближе порога,
то фото считается матчем
```

## Пример

Пусть расстояния от одного лица до селфи:

```text
[0.62, 0.48, 0.31, 0.57]
```

Минимум:

```text
0.31
```

Если порог:

```text
τ = 0.40
```

совпадение есть.

Если порог:

```text
τ = 0.25
```

совпадения уже нет.

# 23. Как это выглядит одной математической схемой

Пусть:

- `f_1, ..., f_M` — лица на текущем фото;
- `s_1, ..., s_N` — лица из селфи-базы;
- `E(·)` — функция извлечения embedding.

Тогда:

## Шаг 1. Embedding каждого лица на фото

```text
a_i = E(f_i) ∈ ℝ^128
```

## Шаг 2. Embedding каждого селфи

```text
b_j = E(s_j) ∈ ℝ^128
```

## Шаг 3. Попарные расстояния

```text
d_{ij} = ||a_i - b_j||
```

## Шаг 4. Ближайшее селфи для каждого лица

```text
d_i* = min_j d_{ij}
```

## Шаг 5. Решение

```text
если существует i такое, что d_i* ≤ τ,
то фото считается найденным
```

||a||

Это центральная операция быстрого сравнения.

# 27. Что можно было бы докрутить математически

Это уже не описание, а инженерный взгляд со стороны.

## Нормализация embeddings

Сейчас сравнение идёт по L2.

Можно было бы проверить вариант с нормализацией:

```text
x̂ = x / ||x||
```

и cosine similarity:

```text
cos(a, b) = (a · b) / (||a|| ||b||)
```

Не факт, что будет лучше именно для этой модели, но это типичный альтернативный маршрут.

## Более строгий spectral metric

FFT-блок сейчас эвристический.  
Если захочется академичнее, можно считать энергию в явной кольцевой high-frequency зоне спектра.

## Более устойчивое правило матча

Сейчас правило по сути такое:

```text
один хороший ближайший сосед ниже порога → матч
```

Иногда полезно добавить:
- top-k,
- margin до второго соседа,
- среднее по нескольким ближайшим.

Но это уже другая логика принятия решения.

# Итог

Главная математика файла — очень здравая и приземлённая:

- изображение сначала проходит через набор дешёвых и средних по цене метрик качества;
- потом каждое лицо представляется как вектор в `ℝ^128`;
- затем все нужные расстояния считаются через формулу

```text
||a - b||² = ||a||² + ||b||² - 2(a · b)
```

- после этого берётся ближайшее селфи;
- и решение принимается обычным порогом.

То есть по сути это аккуратный **vector matching pipeline** с CV-фильтрацией на входе.
